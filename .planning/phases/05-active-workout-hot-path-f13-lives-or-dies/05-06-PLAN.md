## Phase Goal

**As a** Swedish-locale FitnessMaxxing user logging a fractional-plate weight (e.g., 102.5 kg), **I want to** type the decimal separator my keyboard renders (',') OR the period ('.') and have either work, **so that** I can record half-kg increments per the recipe table — not just integer kg.

<objective>
Close the MAJOR UX gap where the weight input rejects decimal separators on Swedish-locale iOS keyboards. UAT 2026-05-13 Phase 3 (Test 3) reported "cannot type a decimal separator at all on iOS keyboard — neither '.' nor ',' is enterable". The schema explicitly allows `multipleOf(0.25)` per D-15.

Investigation finding (from this plan's pre-read of workout/[sessionId].tsx line 684): `keyboardType="decimal-pad"` is ALREADY set per D-11. CONTEXT.md D-09 convention is honored. This means the bug is NOT a missing keyboard prop — Gap #3's first sub-hypothesis ("keyboardType='numeric' instead of 'decimal-pad'") is REFUTED by the codebase.

The actual root cause: on a Swedish-locale iPhone, `keyboardType="decimal-pad"` renders ',' as the separator key — but JavaScript's `Number(',5')` returns `NaN` because the JS parser only recognizes '.'. The user types ',' → onChangeText receives the string ",5" or "102,5" → `z.coerce.number()` calls `Number(...)` → returns `NaN` → Zod fails with "Vikt krävs" (the `error` parameter on the schema), but the user sees a generic "input didn't accept my key" experience because the value never makes it into the controlled state.

The fix is a single `z.preprocess` wrapper on `weight_kg` in `setFormSchema` that replaces ',' with '.' BEFORE `z.coerce.number()` runs. No keyboard change is needed (D-11's `decimal-pad` stays).

**Scope note:** Empty-string handling (`weight_kg: ""`) is OUT OF SCOPE for this plan. Today's behavior is `z.coerce.number()` on "" → 0 → `min(0)` accepts. With or without this plan's preprocess, that pre-existing behavior is unchanged. The plan-checker raised whether the original truth's "empty string fails 'Vikt krävs'" clause matched current behavior — it did not. Empty-string handling is a separate Linear issue (deferred from this plan; create only if the user reports unwanted "accept 0 weight on empty submit" as a separate bug). This plan focuses exclusively on the Swedish-locale comma-normalization fix per Gap #3.

This plan does NOT modify the workout screen (`app/app/(app)/workout/[sessionId].tsx`) because the keyboardType is already correct. If on-device manual UAT after this fix proves the keyboard prop is wrong on the OLD pre-D-11 form path (unlikely — D-09/D-11 say decimal-pad), Task 3 below documents how to extend the fix; but the planned default is schema-only.

Purpose: Restore F6's "logga set i ≤3 sekunder per set" promise for Swedish-locale users. Forced-integer-input is a silent capability regression for the dominant V1 user (the project lives at `Sverige-only V1` per PROJECT.md).

Output: A z.preprocess wrapper on setFormSchema.weight_kg + three new test cases in test-set-schemas.ts + on-device verification (recorded in `human_verification` frontmatter block).

Zod 4 z.preprocess shape:
```typescript
z.preprocess(
  (val) => (typeof val === "string" ? val.replace(/,/g, ".") : val),
  z.coerce.number({ error: "Vikt krävs" }).min(0).max(500).multipleOf(0.25)
)
```
Per Zod v4 docs: `z.preprocess(fn, schema)` runs `fn` first, then pipes its return value into `schema`. Both `z.input` and `z.output` types resolve through the preprocess transform — `z.input` is `unknown` (preprocess can accept anything), `z.output` is `number`. This matches the RHF 3-generic pattern (TFieldValues=input, TContext=undefined, TTransformedValues=output) already used per Phase 4 D-11.

Whitespace handling: typing "102," (trailing comma — common mid-typing state) becomes "102." after replace, which `Number(...)` evaluates to 102 (not NaN — JS is forgiving here). multipleOf(0.25) accepts 102 → PASS. This is fine — the user can submit mid-typed strings; if they meant 102.5 and forgot the trailing 5, the resulting log is 102 kg which is a legit value. No edge-case handling needed.

Multi-comma string "102,5,5" → after `.replace(/,/g, ".")`: "102.5.5" → `Number(...)` returns NaN → Zod rejects. This is the correct behavior (the regex form with `g` flag is required — plain `.replace(",", ".")` would only replace the first comma).
