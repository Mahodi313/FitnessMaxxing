## Phase Goal

**As a** FitnessMaxxing user who just force-quit Expo Go mid-workout and re-opened it offline, **I want to** see my previously-logged sets within ~500ms of tapping Återuppta — not stare at empty exercise cards for 1-3 seconds, **so that** I trust the offline guarantee instead of believing my sets were lost.

<objective>
Close the MAJOR UX gap where the workout screen renders BEFORE the TanStack persister finishes rehydrating from AsyncStorage. F13 data-integrity contract HOLDS — the sets are preserved — but the user perceives data loss during the hydration window, which destroys trust in the offline guarantee.

The current architecture (Phase 4 D-01): `persistQueryClient({ queryClient, persister: asyncStoragePersister, maxAge: 24h })` runs as a SIDE-EFFECT in `app/lib/query/persister.ts` at module load. There is no React-side signal that hydration has COMPLETED. `app/app/_layout.tsx` wraps the tree in a plain `<QueryClientProvider client={queryClient}>` (line 125), NOT a `<PersistQueryClientProvider>` — so no `onSuccess` callback fires when hydration finishes; queries render against an empty cache until AsyncStorage round-trips finish.

The fix is structural: gate the workout-screen render on a hydration-ready signal until the persister has restored the cache.

This plan does NOT modify the persistence contract (maxAge, throttleTime, storage adapter all stay as Phase 5 D-25 set them). It does NOT change the F13 brutal-test contract. It does NOT modify the cold-start draft-resume overlay on (tabs)/index.tsx (that path already works correctly — the gap is specifically on /workout/[sessionId].tsx where useSetsForSessionQuery renders empty before hydration completes).

**Implementation choice pre-resolved (Option A — PersistQueryClientProvider):** Per CLAUDE.md the pinned TanStack version is `@tanstack/react-query-persist-client@^5.100.9` (app/package.json line 38). `PersistQueryClientProvider` with `onSuccess` is the canonical v5 pattern and is exported in this version. Pre-committing here halves executor cognitive load — Option B (imperative `onSuccess` callback on `persistQueryClient(...)`) was a deliberation branch in the prior revision and is dropped.

Purpose: Restore the trust contract. F13 promised "alla 25 set överlever" — the user must perceive that contract, not just verify it post-hoc in Supabase Studio.

Output: A Zustand `usePersistenceStore` signalling hydrated-state; a `<PersistQueryClientProvider>` wrapper in `_layout.tsx` whose `onSuccess` flips the store flag; a render gate on /workout/[sessionId].tsx that shows "Återställer pass…" until the signal flips, then renders WorkoutBody normally.
</objective>
