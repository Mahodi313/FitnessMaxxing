<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Generated types ↔ Hand-edited types | `app/types/database.ts` is generated; hand-editing creates drift between deployed schema and TS types (PITFALLS 3.5). The "do not hand-edit" rule is a discipline boundary. |
| Typed client ↔ Untyped client | `createClient<Database>` enforces compile-time table-name + column-shape correctness. The bare `createClient(...)` return type is `SupabaseClient<any, "public", any>` — any string passes `.from(...)`, any column passes `.select()`. Phase 2's typed pivot makes the type system part of the access-control story (catches typos that would otherwise hit runtime RLS). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-16 | T (Tampering) | Hand-edits to `app/types/database.ts` drifting from deployed schema | mitigate | "Do not hand-edit" rule documented in PATTERNS.md and RESEARCH §"Anti-Patterns". Mitigation is process-level (D-18 convention added in Plan 06: "`npm run gen:types` runs after every schema migration; the generated `app/types/database.ts` is committed in the same commit as the migration"). |
| T-02-17 | I (Information disclosure) | `phase1ConnectTest` referenced a non-existent table; deletion eliminates a vestigial dev-only path | accept | Smoke test was Phase 1 only; its removal is hygiene, not a security event. The replacement (Plan 05's real cross-user test) is Node-side and never bundles. |
| T-02-18 | T | Type-pipeline drift between gen-types output and migration source | mitigate | Plan 03's `db diff` already proved zero drift between local migration and remote schema. Plan 04 generates types FROM that same remote schema — types are consistent by construction. Acceptance criterion `! grep -q "is_warmup"` on the generated types is the cross-check that proves end-to-end (file → push → gen → consume) consistency. |

**ASVS L1 verdict for Plan 04:** No new threat surface introduced beyond what Phase 1's typed client already exposed. The `<Database>` type parameter is purely additive in the security model — it can only ADD compile-time errors, never remove existing runtime checks.
