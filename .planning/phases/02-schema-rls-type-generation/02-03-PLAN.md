<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local migration file ↔ Remote database | The push transfers SQL state across the dev-machine ↔ Supabase boundary using the DB password (server-side credential). Plan 01 collected this credential; this plan consumes it. |
| Remote schema ↔ Studio drift | After `db push`, future Studio edits create a divergence between the committed migration and the live DB. PITFALLS 4.2 explicitly identifies this; D-18 codifies "Studio is read-only" to prevent recurrence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-12 | T (Tampering) | Studio drift between push and next migration | mitigate | `npx supabase db diff` post-push verifies zero divergence at deploy time. D-18 convention (added in Plan 06) makes "Studio is read-only" a durable rule. PITFALLS 4.2 cited in CLAUDE.md / PROJECT.md. |
| T-02-13 | R (Repudiation) | Migration applied without committed source | accept | The migration source IS in repo (committed in Plan 02). The CLI's `db push` records the migration in `supabase_migrations.schema_migrations` on the remote; Studio displays this. Audit trail = git log + remote migration table. |
| T-02-14 | E (Elevation of privilege) | DB password leakage via shell history during interactive `db push` prompt | accept | Password is entered at an interactive CLI prompt (not as a CLI argument), so PowerShell command history does NOT capture it. The token-via-`supabase login` path (Plan 01) is preferred for `gen:types`; for `db push` specifically, interactive password entry is the safest. Out of scope for further mitigation in solo V1. |
| T-02-15 | T | `supabase db reset --linked` accidental destructive run | mitigate | Anti-pattern explicitly listed in RESEARCH §"Anti-Patterns". This task uses ONLY `db push` and `db diff` — both non-destructive. The acceptance criteria do not include `db reset`. |

**ASVS L1 verdict for Plan 03:** V4 + V14 carry over from Plan 02. Push is a one-shot deploy; no new attack surface introduced.
</threat_model>
