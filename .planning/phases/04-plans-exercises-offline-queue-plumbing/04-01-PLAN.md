## RESEARCH §3 + §5 — load-bearing technical contracts

- **Module-load order in `_layout.tsx`**: import `'@/lib/query/client'` FIRST (registers all 8 setMutationDefaults), then `'@/lib/query/persister'` (hydrates from AsyncStorage), then `'@/lib/query/network'` (wires NetInfo + AppState + resume subscription).
- **Two-phase reorder algorithm** to avoid `unique (plan_id, order_index)` violation:
  1. Snapshot cache once.
  2. For each changed row: mutate `order_index = -(idx + 1)` (negative = guaranteed-unused range).
  3. For each changed row: mutate `order_index = newIndex` (final position).
  All carry `scope: { id: \`plan:\${planId}\` }` — TanStack v5 enforces serial replay within a scope.
- **Postgres `int` allows negative values** — verified: no CHECK constraint at `0001_initial_schema.sql:54`. The two-phase trick is safe. (Wave 0 `test-reorder-constraint.ts` proves this.)
- **Upsert vs Insert** — every CREATE uses `.upsert(values, { onConflict: 'id', ignoreDuplicates: true })`. Replay of an already-committed mutation is a no-op (PG `ON CONFLICT DO NOTHING`).
- **Scope chaining** for picker-create-and-add: `useCreateExercise` needs `meta.scopeOverride` so when called from the picker it carries `scope.id = 'plan:<planId>'` (so the chained `useAddExerciseToPlan` runs after the create, preserving FK ordering).
