# Phase 4 — UI Design Contract

> Visual and interaction contract for the (tabs) skeleton (Planer / Historik / Inställningar), plan create/edit/list surfaces, exercise-add sheet, and the global offline-banner. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

> **shadcn note:** shadcn/ui is a React-DOM component library and is incompatible with React Native runtimes. This project uses NativeWind 4 (Tailwind 3 class names compiled to RN styles) per the locked stack in CLAUDE.md and per Phase 3 UI-SPEC. The shadcn initialization gate is therefore **not applicable** and `Tool: none` below is the correct value, not a missed opportunity. All UI is composed from `react-native` core primitives plus `react-native-draggable-flatlist` (CONTEXT.md D-08).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (NativeWind 4 — shadcn is React-DOM only and incompatible with React Native) |
| Preset | not applicable |
| Component library | none — primitives composed from `react-native` (`View`, `Text`, `TextInput`, `Pressable`, `ScrollView`, `KeyboardAvoidingView`, `SafeAreaView`) styled with NativeWind classes; `react-native-draggable-flatlist` for plan_exercises drag-to-reorder (CONTEXT.md D-08) |
| Icon library | `@expo/vector-icons` Ionicons (already installed; CONTEXT.md D-18 — Ionicons explicitly chosen over MaterialCommunityIcons / SF Symbols for consistency with Expo scaffold and offline-availability) |
| Font | System default (San Francisco on iOS via `react-native`); no custom font loaded — Phase 1 did not register any `expo-font` asset; this stays unchanged through Phase 4 |

**Tailwind config inheritance:** `app/tailwind.config.js` uses `darkMode: 'class'` with empty `theme.extend` (verified). All Tailwind v3 default tokens (color palette, spacing scale, fontSize scale) are available unmodified. The class names below resolve against that vanilla palette and remain identical to Phase 3 UI-SPEC for visual continuity.

**New library entering Phase 4:** `react-native-draggable-flatlist` (CONTEXT.md D-08). Its visual surface area is limited to (a) the draggable row container in `plans/[id].tsx`, (b) the per-row drag-handle icon. Both are styled with NativeWind classes — the library does not impose a visual theme.

## Spacing Scale

Declared values (all multiples of 4, mapped to Tailwind v3 default spacing tokens). Identical scale to Phase 3 UI-SPEC for cross-phase visual continuity.

| Token | Value | Tailwind class | Usage in Phase 4 |
|-------|-------|----------------|------------------|
| xs | 4px | `p-1`, `gap-1` | Inline gap between tab-icon and tab-label inside tab-bar; gap inside multi-line target chip ("3×8–12") |
| sm | 8px | `p-2`, `gap-2`, `mt-2` | Field-label-to-input gap; gap between drag-handle and exercise-name inside a plan_exercise row; OfflineBanner ✕ close button hit-padding |
| md | 16px | `p-4`, `gap-4`, `mt-4` | Default horizontal screen padding (matches Phase 3); vertical rhythm between form fields; padding inside plan-list-row card; OfflineBanner inner padding |
| lg | 24px | `p-6`, `gap-6`, `mt-6` | Vertical rhythm between heading block and field block on plan-create / plan-edit; gap between empty-state icon and CTA |
| xl | 32px | `p-8`, `gap-8`, `mt-8` | Vertical separation between major sections on plan-detail (plan-meta block ↔ exercises list ↔ "Lägg till övning" CTA); empty-state vertical centering pad |
| 2xl | 48px | `p-12`, `mt-12` | Top padding inside scrollable plan-create form above heading on unscrolled state (matches Phase 3 form rhythm) |
| 3xl | 64px | `pt-16` | Reserved — not used in Phase 4 surfaces |

**Touch target minimum:** All interactive elements ≥ 44×44pt per Apple HIG (matches Phase 3 contract).
- Primary CTAs ("Skapa plan", "Spara", "Lägg till övning"): `py-4` (16px) + `text-base` (16px) → ~48pt total. Same as Phase 3 sign-up/sign-in.
- Tab-bar items: default Expo Router `<Tabs>` height (~49pt iOS) is HIG-compliant; do not override `tabBarStyle` height.
- Plan-list rows: minimum `py-4` row padding so the entire row is a 44pt+ tap target. The arrow/chevron on the right is decorative — the whole row is `Pressable`.
- Drag-handle: wraps a `<MaterialCommunityIcons name="drag-horizontal-variant">` (or Ionicons `reorder-three-outline`) inside a `Pressable` of `p-3` (12px on each side around a 24pt icon → 48pt total) so long-press lands reliably.
- OfflineBanner ✕ close button: `px-2 py-1` plus `hitSlop={8}` per Phase 3 quick-task convention (commit 4af7462).

Exceptions: none — all values are 4-multiples.

## Color

60/30/10 split mapped to NativeWind classes that auto-flip in dark mode (F15 convention, Phase 1). Identical color contract to Phase 3 UI-SPEC for visual continuity, with one **additional** semantic role (warning/info) introduced for the OfflineBanner.

| Role | Light value | Dark value | Tailwind class pair | Usage |
|------|-------------|------------|----------------------|-------|
| Dominant (60%) | `#FFFFFF` (white) | `#111827` (gray-900) | `bg-white dark:bg-gray-900` | Screen background of all (tabs) screens, plan-create, plan-detail, exercise-add sheet — matches Phase 3 |
| Secondary (30%) | `#F3F4F6` (gray-100) | `#1F2937` (gray-800) | `bg-gray-100 dark:bg-gray-800` | TextInput field background; plan-list row card background; plan_exercise row background inside draggable list; exercise-list row in exercise-add sheet; tab-bar background (`tabBarStyle.backgroundColor`) |
| Accent (10%) | `#2563EB` (blue-600) | `#60A5FA` (blue-400) | `bg-blue-600 dark:bg-blue-500` (button bg) + `text-blue-600 dark:text-blue-400` (link text + active tab tint) | **Reserved for: see "Accent reserved for" list below** |
| Destructive | `#DC2626` (red-600) | `#F87171` (red-400) | `text-red-600 dark:text-red-400` / `bg-red-600 dark:bg-red-500` | Inline form-field error messages (text); "Arkivera plan" menu-item label (text only); confirmation dialog destructive button (background) — see Copywriting Contract |
| Warning/Info | `#FEF3C7` bg + `#92400E` text (light); `#78350F` bg + `#FEF3C7` text (dark) | (same) | `bg-yellow-100 dark:bg-yellow-900` + `text-yellow-900 dark:text-yellow-100` | **OfflineBanner only.** Yellow signals "system condition affecting your work" without escalating to red (which is reserved for errors and destructive actions). New role added in Phase 4 — not present in Phase 3 |
| Body text | `#111827` (gray-900) | `#F9FAFB` (gray-50) | `text-gray-900 dark:text-gray-50` | All non-error body and label text; field input text; tab-bar inactive tint |
| Muted text | `#6B7280` (gray-500) | `#9CA3AF` (gray-400) | `text-gray-500 dark:text-gray-400` | TextInput placeholder text; plan-row description (one-line truncated); empty-state body copy; placeholder-tab copy ("Historik kommer i Phase 6"); drag-handle icon tint when row is at-rest |

**Accent reserved for:**
1. Primary CTA button background ("Skapa plan", "Spara", "Lägg till övning", "Skapa övning")
2. Active tab tint (`tabBarActiveTintColor: '#2563EB'` light / `'#60A5FA'` dark — bound via `useColorScheme()` in `(tabs)/_layout.tsx`)
3. TextInput border when the field is focused (`focus:border-blue-600 dark:focus:border-blue-500` — same as Phase 3)
4. Empty-state icon (Ionicons `barbell-outline` tinted with accent so the empty Planer-tabb has one focal point)
5. Inline navigation/text-link affordances inside plan-detail (e.g. "Visa arkiverade planer" if/when added — currently no such link in Phase 4)

**Accent must NOT be used for:**
- Field labels (those use Body color)
- Headings (those use Body color, weight differentiates them)
- Borders of unfocused fields (those use `border-gray-300 dark:border-gray-700`)
- Tab-bar inactive icons (those use Muted text color → `tabBarInactiveTintColor: '#6B7280'` light / `'#9CA3AF'` dark)
- Drag-handle icon (Muted text color)
- Plan-list-row chevron (Muted text color — decorative)
- The OfflineBanner background (Warning/Info role owns that surface)

**Destructive color is used in two places in Phase 4:**
1. **Inline form errors** — text only, on white/gray background. Identical to Phase 3.
2. **"Arkivera plan" action** — the menu-item label text uses `text-red-600 dark:text-red-400` to telegraph that the action removes the plan from the list. The confirmation dialog's "Arkivera"-button background uses `bg-red-600 dark:bg-red-500` with white text. (Archive is reversible at the data level — `archived_at` is just a timestamp set/unset — but irreversible at the UI level in V1 because no restore-flow exists per CONTEXT.md `<deferred>`. The red affordance correctly signals "this is the destructive-feeling path".)

**Status bar contrast:** `<StatusBar style="auto" />` from Phase 1 stays. It flips status-bar-icon color with system theme so icons contrast against `bg-white` (light) and `bg-gray-900` (dark). Do not switch to `style="light"` or `style="dark"` for any Phase 4 surface.

**OfflineBanner color rationale:** Yellow (warning/info) chosen over red (destructive) because being offline is an expected, recoverable system state in an offline-first app — not a user error and not a destructive action. Red would create false urgency and erode the "your data is safe offline" message the banner is supposed to communicate. Contrast: `text-yellow-900` on `bg-yellow-100` = 8.59:1 (AAA); `text-yellow-100` on `bg-yellow-900` = 11.7:1 (AAA).

### Exercise-add sheet (triggered from `plans/[id].tsx` "Lägg till övning")

**Sheet presentation decision (CONTEXT.md Discretion area):** **Expo Router `presentation: 'modal'`** (full-screen modal that slides up from the bottom on iOS). Rationale:
- (a) No new dependency required (`@gorhom/bottom-sheet` and `react-native-actions-sheet` were the alternatives).
- (b) iOS standard modal slides over the full screen, includes its own "Avbryt" header automatically, and dismisses with a downward swipe — all out-of-the-box behavior the user already knows.
- (c) Phase 4 success #4 (airplane-mode-test) does not require any sheet-specific UX; modal route is the lowest-risk, lowest-dep choice.
- (d) Bottom-sheet (drag-handle, peek-state) is V1.1 polish if soak-test reveals friction.

Implementation: file at `app/app/(app)/plans/[id]/exercise-picker.tsx` with `<Stack.Screen options={{ presentation: 'modal', title: 'Lägg till övning' }} />`. Modal owns its own `<Stack.Screen>` so the parent stack header on plan-detail is not affected.

**Exercise search implementation decision (CONTEXT.md Discretion area):** **Client-side `.filter()`** on the cached `useExercisesQuery()` result. Rationale:
- (a) V1 is single-user, "egna övningar" only (CONTEXT.md Discretion confirmed: `exercises` LIST scope = `user_id = (select auth.uid())`).
- (b) Realistic personal data set is 20–80 exercises in V1 — a single in-memory `.filter(e => e.name.toLowerCase().includes(q.toLowerCase()))` is O(n) and runs in <1ms on iPhone.
- (c) Server-side `.ilike('%q%')` would invalidate the offline-first contract — search would break in airplane mode.
- (d) Cache is already hydrated via `usePlansQuery` parent context.
- V1.1 may add server-side fuzzy match if global library (F20, V2) lands.

### Destructive confirmation (only one in Phase 4 per CONTEXT.md D-12)

| Action | Trigger | Confirmation copy |
|--------|---------|-------------------|
| Archive plan | Tap "Arkivera plan" in plan-detail overflow menu | iOS native `Alert.alert()`:<br>**Title:** `Arkivera "{plan.name}"?`<br>**Message:** `Planen tas bort från listan. Pass som använt planen behåller sin historik.`<br>**Buttons:** `Avbryt` (default, light text) / `Arkivera` (destructive style — `style: 'destructive'` on iOS renders red) |

**Why iOS native `Alert.alert` and not a custom modal:**
- (a) Native alert respects iOS Dynamic Type, dark mode, and VoiceOver out-of-box.
- (b) "Arkivera plan" is a single low-frequency action — building a custom destructive-modal component for one use site is over-engineered for V1.
- (c) Native alert correctly applies destructive button styling without us having to maintain that color contract.
- (d) Cancel button order: iOS default is destructive on the right. We keep the iOS default — do not reverse for safety bias because the action is recoverable at the data level.

**No destructive confirmation for:**

### Loading / cold-start

The TanStack persister hydrates from AsyncStorage synchronously enough that this indicator typically does not render in steady-state use; it exists for first-launch + post-clear paths.
- **Per-mutation loading:** Optimistic updates flip the cache immediately, so the user **never sees a spinner during a mutation.** The OfflineBanner communicates "queued for sync" implicitly via "Du är offline — ändringar synkar när nätet är tillbaka." When online, mutations are typically <300ms and the cache is already showing the optimistic state.

- **Mutation behavior offline:** Per `networkMode: 'offlineFirst'` + `setMutationDefaults`, mutations queue and the optimistic-update visually shows the row as if it succeeded. **No "queued" badge per row** — F24 sync-state-badge is V2-deferred (CONTEXT.md `<deferred>` + REQUIREMENTS.md).
