# Phase 3 — UI Design Contract

> Visual and interaction contract for `(auth)/sign-up.tsx`, `(auth)/sign-in.tsx`, and the temporary post-login surface in `(app)/index.tsx`. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **shadcn note:** shadcn/ui is a React-DOM component library and is incompatible with React Native runtimes. This project uses NativeWind 4 (Tailwind 3 class names compiled to RN styles) per the locked stack in CLAUDE.md. The shadcn initialization gate is therefore **not applicable** and `Tool: none` below is the correct value, not a missed opportunity.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (NativeWind 4 — shadcn is React-DOM only and incompatible with React Native) |
| Preset | not applicable |
| Component library | none — primitives composed from `react-native` (`View`, `Text`, `TextInput`, `Pressable`) styled with NativeWind classes |
| Icon library | `@expo/vector-icons` (already in `package.json`) — used sparingly in Phase 3 (no icons required for the two forms; a single optional eye-toggle for the password field is allowed if implemented) |
| Font | System default (San Francisco on iOS via `react-native`); no custom font loaded in V1 — Phase 1 did not register any `expo-font` asset |

**Tailwind config inheritance:** `app/tailwind.config.js` uses `darkMode: 'class'` and an empty `theme.extend`, so all Tailwind v3 default tokens (color palette, spacing scale, fontSize scale) are available unmodified. The class names referenced below resolve against that vanilla palette.

## Spacing Scale

Declared values (all multiples of 4, mapped to Tailwind v3 default spacing tokens):

| Token | Value | Tailwind class | Usage in Phase 3 |
|-------|-------|----------------|------------------|
| xs | 4px | `p-1`, `gap-1` | Inline label-to-helper-text gap |
| sm | 8px | `p-2`, `gap-2`, `mt-2` | Compact gaps inside form rows; field-error-message margin-top |
| md | 16px | `p-4`, `gap-4`, `mt-4` | Default form field vertical rhythm; horizontal screen padding |
| lg | 24px | `p-6`, `gap-6`, `mt-6` | Section padding between heading block and field block |
| xl | 32px | `p-8`, `gap-8`, `mt-8` | Vertical separation between primary form and secondary nav link ("Har du redan ett konto? Logga in") |
| 2xl | 48px | `p-12`, `mt-12` | Top padding inside form container above heading on the unscrolled state |
| 3xl | 64px | `pt-16` | Reserved — not used in Phase 3 forms |

**Touch target minimum:** Submit button and navigation links must hit at minimum 44×44pt per Apple HIG. Use `py-3` (12px vertical) + `text-base` (16px) for buttons → 16 + 12 + 12 = 40pt; bump to `py-4` (16px) for the primary CTA so total height is ~48pt. Plain text nav links ("Logga in"/"Registrera") wrap a `Pressable` with `py-3 px-2` so the tap target reaches 44pt vertical even when the visible text is smaller.

Exceptions: none — all values are 4-multiples.

## Typography

Tailwind v3 default `fontSize` scale, restricted to four roles for Phase 3:

| Role | Size | Tailwind class | Weight | Tailwind weight class | Line Height |
|------|------|----------------|--------|------------------------|-------------|
| Body | 16px | `text-base` | 400 (regular) | `font-normal` (default) | 1.5 (Tailwind `text-base` ships `lh-6` = 24px = 1.5×) |
| Label | 14px | `text-sm` | 600 (semibold) | `font-semibold` | 1.43 (Tailwind `text-sm` ships `lh-5` = 20px) |
| Heading | 24px | `text-2xl` | 600 (semibold) | `font-semibold` | 1.33 (Tailwind `text-2xl` ships `lh-8` = 32px) |
| Display | 30px | `text-3xl` | 600 (semibold) | `font-semibold` | 1.2 (Tailwind `text-3xl` ships `lh-9` = 36px) |

**Two weights only:** `font-normal` (400) for body and field input text; `font-semibold` (600) for labels, headings, the display, and the primary CTA. No `font-bold` (700) and no `font-medium` (500) — keeps the contract to two weights per spec quality dimension.

**Per-screen role mapping:**
- `(auth)/sign-up.tsx` heading: "Skapa konto" → Display (`text-3xl font-semibold`)
- `(auth)/sign-in.tsx` heading: "Logga in" → Display (`text-3xl font-semibold`)
- Form field labels ("Email", "Lösen", "Bekräfta lösen") → Label (`text-sm font-semibold`)
- Form field input text + error messages + nav-link copy → Body (`text-base font-normal`); error messages add color override `text-red-600 dark:text-red-400` only (size and weight unchanged)
- Primary CTA button label ("Skapa konto" / "Logga in") → Body size with semibold (`text-base font-semibold`)
- Heading reserved for future per-section headings inside `(app)` surface (not exercised in Phase 3)

## Color

60/30/10 split mapped to NativeWind classes that auto-flip in dark mode (per F15 convention from Phase 1):

| Role | Light value | Dark value | Tailwind class pair | Usage |
|------|-------------|------------|----------------------|-------|
| Dominant (60%) | `#FFFFFF` (white) | `#111827` (gray-900) | `bg-white dark:bg-gray-900` | Screen background of `(auth)/sign-up`, `(auth)/sign-in`, `(app)/index` — matches Phase 1 smoke-test convention exactly |
| Secondary (30%) | `#F3F4F6` (gray-100) | `#1F2937` (gray-800) | `bg-gray-100 dark:bg-gray-800` | TextInput field background; the temporary sign-out card container in `(app)/index.tsx` |
| Accent (10%) | `#2563EB` (blue-600) | `#60A5FA` (blue-400) | `bg-blue-600 dark:bg-blue-500` (button bg) + `text-blue-600 dark:text-blue-400` (link text) | **Reserved for: (1) the primary CTA button background on sign-up/sign-in submit, (2) the inline navigation link text ("Har du redan ett konto? Logga in" / "Skapa konto"), (3) focused TextInput border. Nothing else.** |
| Destructive | `#DC2626` (red-600) | `#F87171` (red-400) | `text-red-600 dark:text-red-400` | Inline form-field error messages only; no destructive *actions* exist in Phase 3 |
| Body text | `#111827` (gray-900) | `#F9FAFB` (gray-50) | `text-gray-900 dark:text-gray-50` | All non-error body and label text; field input text |
| Muted text | `#6B7280` (gray-500) | `#9CA3AF` (gray-400) | `text-gray-500 dark:text-gray-400` | TextInput placeholder text; field-helper-text ("Minst 12 tecken") if rendered as helper instead of error |

**Accent reserved for:**
1. Primary CTA button background (sign-up submit, sign-in submit, sign-out button)
2. In-screen navigation link text between auth screens ("Har du redan ett konto? Logga in" on sign-up; "Inget konto? Registrera" on sign-in)
3. TextInput border when the field is focused (`border-blue-600 dark:border-blue-500`)

**Accent must NOT be used for:**
- Field labels (those use Body color)
- Headings (those use Body color, weight differentiates them)
- Borders of unfocused fields (those use `border-gray-300 dark:border-gray-700`)
- Decorative dividers
- The placeholder smoke-test "Hello {email}" greeting in `(app)/index.tsx`

**Destructive color is read-only in Phase 3:** Inline Zod validation errors render in red but no Phase 3 action is destructive (sign-out is reversible by signing in again). Phase 7 owns destructive confirmation patterns when account-deletion lands.

**Status bar contrast:** `<StatusBar style="auto" />` from Phase 1 stays — it flips status-bar-icon color with system theme so icons contrast against `bg-white` in light and `bg-gray-900` in dark. Do not switch to `style="light"` or `style="dark"` for auth screens.

All copy is in Swedish per PROJECT.md (svensk användare, svensk app) and CONTEXT.md D-15.

### Form-level copy

| Element | Screen | Copy |
|---------|--------|------|
| Heading | sign-up | `Skapa konto` |
| Heading | sign-in | `Logga in` |
| Field label — email | both | `Email` |
| Field label — password | both | `Lösen` |
| Field label — confirm password | sign-up | `Bekräfta lösen` |
| Field placeholder — email | both | `du@example.com` |
| Field placeholder — password | both | (empty — `secureTextEntry` masks input; no placeholder needed) |
| Helper text — password | sign-up | `Minst 12 tecken` (rendered below the password field as muted Body text **before** validation fires; replaced by inline error if Zod fails) |
| Primary CTA — sign-up | sign-up | `Skapa konto` |
| Primary CTA — sign-in | sign-in | `Logga in` |
| Primary CTA loading state | both | `Loggar in…` (sign-in) / `Skapar konto…` (sign-up) — button stays full-width, label swaps; button is disabled with `opacity-60` while RHF `isSubmitting` is true |

### Error states (inline, below the field, red, Body size, no icons)

| Trigger | Copy |
|---------|------|
| Email format invalid (Zod `z.string().email()`) | `Email måste vara giltigt` |
| Email empty (Zod `.min(1)`) | `Email krävs` |
| Password too short on sign-up (Zod `.min(12)`) | `Minst 12 tecken` |
| Password empty on sign-in (Zod `.min(1)`) | `Lösen krävs` |
| Confirm-password mismatch (Zod `.refine`) | `Lösen matchar inte` |
| Supabase error: duplicate email on sign-up | `Detta email är redan registrerat — försök logga in` (rendered under the email field per CONTEXT.md D-03) |
| Supabase error: invalid credentials on sign-in | `Fel email eller lösen` (rendered under the password field — generic per ASVS V2.1.4 to avoid revealing which field is wrong) |
| Supabase error: network failure (any other thrown error) | `Något gick fel. Försök igen.` (rendered as a screen-level banner above the form, dismissible by tapping) |

### Empty state

Phase 3 has **no empty-state surface in the auth flow** (sign-up and sign-in are always-rendering forms, never empty lists).

The temporary `(app)/index.tsx` post-login surface is also not an "empty state" — it is a placeholder. Its copy:

| Element | Copy |
|---------|------|
| Greeting | `Inloggad som {email}` (uses `useAuthStore(s => s.session?.user.email)`) |
| Heading | `FitnessMaxxing` |
| Body note | `Plan-skapande kommer i nästa fas.` (signals to future-self that this surface is intentionally provisional) |
| Sign-out button label | `Logga ut` |

### Destructive confirmation

Phase 3 has **no destructive actions**. Sign-out is non-destructive (reversible by signing in). Therefore: no confirmation dialog, no `Alert.alert`, no warning text. The sign-out button fires `useAuthStore.getState().signOut()` immediately on tap.

### Error states — additional rules

- Error text appears only after the field has been blurred at least once (RHF `mode: 'onBlur'` per CONTEXT.md Discretion). Errors do not fire on every keystroke.
- Submitting the form via the primary CTA when validation has not yet fired forces all errors to render simultaneously (RHF default behavior — `handleSubmit` triggers `trigger()` first).
- Errors clear automatically as the user corrects the field (RHF + Zod re-validation on change after first submit attempt).

## Visuals (Phase 3 component inventory)

The following primitives compose the two auth screens. Each is a plain `react-native` element styled with the NativeWind classes specified.

### TextInput field (used 5 times: 2 emails + 3 passwords)

```
View (className="gap-2")
  Text label  → text-sm font-semibold text-gray-900 dark:text-gray-50
  TextInput   → w-full rounded-lg border bg-gray-100 dark:bg-gray-800
                px-4 py-3 text-base text-gray-900 dark:text-gray-50
                placeholder:text-gray-500 dark:placeholder:text-gray-400
                Default border:  border-gray-300 dark:border-gray-700
                Focused border:  border-blue-600 dark:border-blue-500
                Error border:    border-red-600 dark:border-red-400
  Text error  → text-base text-red-600 dark:text-red-400  (only when fieldState.error exists)
```

Field-specific props:
- Email field: `keyboardType="email-address"`, `autoCapitalize="none"`, `autoComplete="email"`, `textContentType="emailAddress"`
- Password field: `secureTextEntry={true}`, `autoCapitalize="none"`, `autoComplete="password"` (sign-in) / `autoComplete="new-password"` (sign-up), `textContentType="password"` / `"newPassword"`
- Confirm-password field: same as password but `autoComplete="new-password"`, `textContentType="newPassword"`

### Primary CTA button

```
Pressable  → w-full rounded-lg bg-blue-600 dark:bg-blue-500
             py-4 items-center justify-center
             disabled:opacity-60  active:opacity-80
  Text     → text-base font-semibold text-white
```

Press feedback: `active:opacity-80` (NativeWind 4 supports the `active:` variant). Disabled while `isSubmitting` is true.

### Inline nav link

```
View (flex-row items-center justify-center mt-8 gap-1)
  Text         → text-base text-gray-900 dark:text-gray-50  ("Har du redan ett konto?")
  Pressable    → py-3 px-2  (44pt tap target)
    Text       → text-base font-semibold text-blue-600 dark:text-blue-400  ("Logga in")
```

### Screen container

```
SafeAreaView  → flex-1 bg-white dark:bg-gray-900
  KeyboardAvoidingView  → flex-1  behavior="padding" (iOS)
    ScrollView          → contentContainerStyle={{flexGrow:1, paddingHorizontal:16, paddingVertical:48}}
                          keyboardShouldPersistTaps="handled"
      View (gap-6)
        View (gap-2)            ← heading block
          Text (display)        ← "Skapa konto" / "Logga in"
        View (gap-4)            ← field block
          [TextInput field × 2 or 3]
        Pressable (primary CTA)
        View (nav link)
```

### Loading / cold-start

Cold-start splash hold is OS-level (`expo-splash-screen` per CONTEXT.md D-04/D-05) — there is **no React-rendered loading screen** in Phase 3. The native splash stays up until `useAuthStore(s => s.status) !== 'loading'`. Once status flips, splash hides and the user sees either the auth screen or the post-login surface, never a "Loading…" spinner in JS.

In-form loading: the primary CTA button label swaps to its `…` variant (`Loggar in…` / `Skapar konto…`) and the button is disabled. No global activity indicator.

### Offline state in auth screens

Per CONTEXT.md Discretion: Phase 3 does **not** wire a NetInfo banner. If `supabase.auth.signUp/signInWithPassword` rejects due to network, the catch-block surfaces the generic banner copy `Något gick fel. Försök igen.` above the form. Phase 4 owns the global offline-banner convention; Phase 3 does not preempt it.

## Interaction Contracts

| Surface | Trigger | Result |
|---------|---------|--------|
| `(auth)/sign-up.tsx` submit | Primary CTA tap with valid form | `supabase.auth.signUp({ email, password })` → `onAuthStateChange` fires → Zustand store flips `status` to `'authenticated'` → `Stack.Protected guard={!!session}` re-evaluates → user sees `(app)/index.tsx` automatically. No imperative `router.replace`. |
| `(auth)/sign-in.tsx` submit | Primary CTA tap with valid form | `supabase.auth.signInWithPassword({ email, password })` → same atomic re-route as above. |
| `(auth)/sign-up.tsx` nav link | Tap on "Logga in" | `router.replace('/(auth)/sign-in')` (replace, not push, so back-button doesn't return to sign-up after creating an account) |
| `(auth)/sign-in.tsx` nav link | Tap on "Registrera" | `router.replace('/(auth)/sign-up')` |
| `(app)/index.tsx` sign-out button | Tap | `useAuthStore.getState().signOut()` → `supabase.auth.signOut()` → `queryClient.clear()` → `onAuthStateChange` flips `status` to `'anonymous'` → `Stack.Protected` re-evaluates → user sees `(auth)/sign-in.tsx`. No confirm dialog (non-destructive). |
| Cold start | App launch | Native iOS splash held by `SplashScreen.preventAutoHideAsync()`; `useEffect` in `RootLayout` calls `SplashScreen.hideAsync()` when `status !== 'loading'`. Authenticated users land directly in `(app)/index.tsx` with zero flicker. |
| Field blur | User leaves a field with invalid input | RHF `mode: 'onBlur'` fires Zod resolver → fieldState.error set → red border + inline error text render. |
| Field re-edit after error | Keystroke after first error shown | RHF re-runs validation on change after first submit attempt → error clears as soon as input becomes valid. |

## Accessibility Floor

| Item | Implementation |
|------|----------------|
| Touch target | All `Pressable` elements ≥ 44×44pt (CTA button is 48pt vertical via `py-4`; nav links wrap text in a `Pressable` with `py-3 px-2`) |
| Color contrast | Body `text-gray-900` on `bg-white` = 15.3:1 (AAA); `text-gray-50` on `bg-gray-900` = 17.4:1 (AAA); CTA `text-white` on `bg-blue-600` = 4.86:1 (AA Large + AA Body); error `text-red-600` on `bg-white` = 4.83:1 (AA Body) |
| Form labels | Every TextInput has a sibling `<Text>` label visually associated above it (gap-2 column) and `accessibilityLabel` matching the visible label string |
| Error association | Each field's error `<Text>` has `accessibilityLiveRegion="polite"` (Android) — iOS VoiceOver picks up the error via re-render announcement |
| Keyboard semantics | Email field uses `keyboardType="email-address"` + `autoCapitalize="none"`; password fields use `secureTextEntry` + `autoComplete` and `textContentType` set so iOS Password Manager and AutoFill work |
| Focus management | After Zod fails on submit, RHF `setFocus(firstErrorField)` runs (RHF default) so VoiceOver lands on the first invalid field |

## Dark Mode Verification (F15 Convention)

Both `(auth)` screens and the temporary `(app)/index.tsx` MUST render correctly in both light and dark mode out-of-box. Verification checklist for the executor:

- [ ] Background flips: `bg-white` ↔ `bg-gray-900`
- [ ] Body text flips: `text-gray-900` ↔ `text-gray-50`
- [ ] Field background flips: `bg-gray-100` ↔ `bg-gray-800`
- [ ] Field border flips: `border-gray-300` ↔ `border-gray-700`
- [ ] CTA flips: `bg-blue-600` ↔ `bg-blue-500`
- [ ] Link text flips: `text-blue-600` ↔ `text-blue-400`
- [ ] Error text flips: `text-red-600` ↔ `text-red-400`
- [ ] Status bar icons remain visible in both modes (handled by `StatusBar style="auto"` from `app/app/_layout.tsx`, untouched in Phase 3)

Manual test: simulator → Settings → Developer → Dark Appearance toggle. Both themes must pass without a reload.
