## Phase Goal

**As a** FitnessMaxxing user navigating back from /workout/[sessionId] to /plans/[id] mid-workout, **I want to** see the ActiveSessionBanner consistently (so I can tap it to return to my pass), **so that** the F8 affordance to resume an active session is always one tap away from any non-workout route.

<objective>
Close the MINOR intermittent gap where ActiveSessionBanner sometimes fails to render on back-nav from /workout/<id> to /plans/<id>. UAT 2026-05-13 Phase 2 side-observation: "sometimes lands on plan-row dialog (plans/[id]) WITHOUT the ActiveSessionBanner module — only the second navigation in/out renders the banner correctly. Intermittent, not 100% reproducible on first try."

The current implementation (Phase 5 D-22, file 05-02 + active-session-banner.tsx 76 LOC):
- `useActiveSessionQuery()` is consumed inside the banner component
- `useSegments()` provides the route segments; the hide-gate at line 49-50 reads `segments.some((s) => s === "workout")` and returns `null` when on /workout/*
- The (tabs)/_layout.tsx mounts the banner between OfflineBanner and Tabs (line 39 per VERIFICATION.md key-link table)
- (tabs)/_layout.tsx uses `freezeOnBlur: true` (Phase 4 D-08 / commit af6930c) — so when user navigates AWAY from a (tabs) screen and BACK, the screen state is preserved
- The banner is mounted INSIDE the (tabs) layout. When user is on /workout/<id> the banner returns null. When user nav-back to /plans/<id> (which is inside (app), NOT inside (tabs)) the banner is STILL NOT rendered because (tabs)/_layout.tsx isn't on the stack — but plans/[id] IS NOT a (tabs) screen, it's an (app) screen ABOVE the tabs.

Re-read this distinction carefully — it changes the diagnosis:
- (tabs) is one navigation group (Planer, Historik, Inställningar)
- /plans/[id] is an (app) route OUTSIDE the (tabs) group (per Phase 4 routing)
- /workout/[sessionId] is also an (app) route OUTSIDE the (tabs) group
- ActiveSessionBanner is mounted inside (tabs)/_layout.tsx — meaning it is ONLY rendered when the user is on Planer / Historik / Inställningar

If the UAT-reported reproduction is "back from /workout/<id> → /plans/<id>" (NOT to (tabs)), the banner SHOULD NOT render on /plans/<id> because /plans/<id> is not a tab. The "intermittent" observation might be the user's expectation mismatch — banner is correctly absent on /plans/<id>.

But the UAT says "only the second navigation in/out renders the banner correctly" — meaning the user nav-in to /workout, back to /plans, then forward to /workout, then back AGAIN to /plans, and the second time the banner appears. That cannot be a (tabs)-vs-(app) routing mismatch; if it were, the banner would never render on /plans/<id> regardless of nav count.

Conclusion: the UAT-reported destination might actually be (tabs)/index.tsx (Planer tab — same name confusion as the plans-detail route /plans/<id>). The hypothesis-tree:
- Hypothesis 1: User actually back-navs to (tabs)/index.tsx (Planer TAB, not the /plans/<id> detail screen), and the banner mounted inside (tabs)/_layout.tsx fails to render on first arrival due to useSegments / useFocusEffect / freezeOnBlur interaction.
- Hypothesis 2: User back-navs to /plans/<id> and expected the banner there too. Resolution: spec clarification — banner is by design only on (tabs).

This plan's Task 1 is investigation-first to disambiguate which of these is the actual user observation, and only then plan the fix.

**Default sub-option:** The DEFAULT planning expectation is **sub-option C — spec clarification (no code change)**, because the banner is by design only mounted inside `(tabs)/_layout.tsx` and therefore correctly absent on every non-(tabs) route including `/plans/<id>`. The other sub-options (A: segments-race fix, B: cache-staleness fix, AB: both) exist for the scenario where the 10-arrival log table proves the destination IS `(tabs)/index.tsx`, hasData=true, onWorkoutRoute=false, AND the banner is missing on arrival.

Purpose: Restore the F8 trust contract — when an active session exists, the resume affordance is consistently available wherever it is supposed to be available. The fix may be a code change (Hypothesis 1) OR a UX clarification (Hypothesis 2); we cannot ship a code change until we know which.

Output: An investigation artifact + a code-or-spec change driven by the investigation finding. NO speculative refactoring of the banner without evidence.
</objective>

```typescript
export function ActiveSessionBanner() {
  const router = useRouter();
  const segments = useSegments();
  const { data: activeSession } = useActiveSessionQuery();
  // ...
  const onWorkoutRoute = (segments as readonly string[]).some((s) => s === "workout");
  if (!activeSession || onWorkoutRoute) return null;
  return <Pressable ...>...</Pressable>;
}
```
Mounted in (tabs)/_layout.tsx between OfflineBanner and Tabs (per VERIFICATION.md).

Routing structure (per file pre-read + Phase 4 routing):
- (app)/_layout.tsx = the (app) Stack
- (app)/(tabs)/_layout.tsx = the Tabs layout for Planer/Historik/Inställningar
- (app)/plans/[id].tsx = pushed onto (app) Stack ABOVE (tabs) — when user is on /plans/<id>, (tabs)/_layout.tsx is NOT in the render tree
- (app)/workout/[sessionId].tsx = pushed onto (app) Stack ABOVE (tabs) — same as above

This means: ActiveSessionBanner ONLY renders when the user is on one of the three tab screens. If the UAT reproduction is "back-nav to /plans/<id>", the banner correctly does not render — that's a spec question, not a bug.

useSegments() behavior:
- Returns the current route's segment array (e.g., `["(app)", "(tabs)", "index"]` for Planer tab, `["(app)", "plans", "[id]"]` for plan-detail, `["(app)", "workout", "[sessionId]"]` for workout)
- Re-fires on every navigation; the banner re-renders on every focus change because it's mounted in (tabs)/_layout.tsx which is itself reactive to navigation

freezeOnBlur (Phase 4 D-08 / commit af6930c):
- (app)/_layout.tsx Stack uses `freezeOnBlur: true` on screen options. When user navigates AWAY from a (tabs) screen to /workout, the (tabs) tree is "frozen" — useSegments may not re-fire there until the user navigates BACK to (tabs).
- When user navigates back to (tabs) — useSegments re-evaluates with the (tabs)-current segments. The banner re-renders. THIS path should work.

useActiveSessionQuery (sessions.ts lines 48-67 per VERIFICATION.md):
- queryKey: sessionsKeys.active()
- queryFn: LIMIT 1 WHERE finished_at IS NULL ORDER BY started_at DESC
- Default staleTime is 30s (from queryClient defaultOptions in client.ts line 71)
- No `refetchOnFocus` / `refetchOnMount` explicit configuration — uses defaults
- The data is populated optimistically via the dual-write onMutate in setMutationDefaults[['session','start']] (per 05-02 SUMMARY); so when the user taps Starta pass, the active-session cache is immediately populated

The race candidates:
- A) useSegments stale on first focus after (tabs) unfreezes — but Expo Router's useSegments is documented as reactive; unlikely
- B) useActiveSessionQuery returns undefined briefly when freezeOnBlur unfreezes and the cache wasn't seeded fresh
- C) The Pressable wrapping the banner has a `mt-2` className (line 57) but no minimum-height — if it renders for a frame with no data then re-renders with data, the layout might shift; this isn't a "missing banner" though, just a flicker
- D) The destination route is NOT a tab — banner correctly absent (spec-not-bug) — this is the DEFAULT expectation per the prose above
