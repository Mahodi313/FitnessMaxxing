## Phase Goal

**As a** FitnessMaxxing user logging sets offline on a physical iPhone, **I want to** trust that every set I log produces exactly one row in the database with a unique `(session_id, exercise_id, set_number)` tuple — even when force-quit + cache rehydration race against my next tap, **so that** my workout history (volume, last-value, future graphs) reflects what I actually did and "får aldrig förlora ett set" extends to "får aldrig duplicera ett set heller."

<objective>
Close the P0 schema gap that allowed 6 silent duplicate rows to land in session 379cfd29-a06f-4dbc-b429-ab273b16c096 during the F13 UAT (2026-05-13). The fix is two-part: (1) add a UNIQUE constraint on `(session_id, exercise_id, set_number)` so Postgres rejects the duplicate at the DB layer regardless of client UUID; (2) move `set_number` assignment server-side via a BEFORE INSERT trigger so the slow-hydration race that produced the duplicates is structurally impossible.

This plan supersedes Phase 5 decision D-16 ("client-side count + 1; race accepted"). The deviation is documented in this plan's frontmatter `deviates_from` block with rationale.

Purpose: Restore the F13 contract integrity. The brutal-test UAT proved the data SURVIVED force-quit, but Phase 9 inspection revealed it ARRIVED corrupted in a prior session. The schema gap is latent — any user, on any device, under sufficient cache-hydration pressure, will reproduce the duplicates.

Output: 3 numbered SQL migrations, an updated `useAddSet` payload contract, an updated optimistic onMutate that produces a provisional set_number for the cache only, a new `test:f13-brutal` npm script invoking the existing `verify-f13-brutal-test.ts` plus a duplicate-detection SQL gate, an extended `test-rls.ts` with a natural-key uniqueness assertion, 3 untracked diagnostic scripts committed, and the folded info-tier nit (unused `Href` import) cleaned.
</objective>
