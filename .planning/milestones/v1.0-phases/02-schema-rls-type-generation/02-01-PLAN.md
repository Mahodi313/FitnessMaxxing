<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer-machine ↔ Supabase remote | DB password, service-role key, personal access token cross this boundary; all are user-side secrets that must reach the remote project but never the JS bundle. |
| Node process ↔ Metro bundler | `app/scripts/` (Node-side, can read service-role) vs `app/lib/`, `app/app/` (Metro-bundled, can ONLY read EXPO_PUBLIC_*). Plan 01 establishes the convention; Plan 05 enforces it. |
| `app/.env.example` (tracked) ↔ `app/.env.local` (gitignored) | Documentation/template vs real secrets. Plan 01 only ever puts placeholder values in tracked files. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | I (Information disclosure) / E (Elevation of privilege) | `SUPABASE_SERVICE_ROLE_KEY` placeholder line in `app/.env.example` | mitigate | Key name has NO `EXPO_PUBLIC_` prefix (PITFALLS 2.3); explicit Swedish warning comment in `.env.example` documents the rule for future operators. Acceptance criterion `grep -c "EXPO_PUBLIC_SUPABASE_SERVICE_ROLE" app/.env.example` = 0 enforces this at plan-checker time. |
| T-02-02 | I / E | `app/.env.local` real service-role value | mitigate | File is gitignored from Phase 1 (Phase 1 D-07); `git check-ignore` is an acceptance criterion; user pastes the value out-of-band (never in chat). |
| T-02-03 | T (Tampering) | `app/supabase/config.toml` project_id binding | accept | project_id is non-sensitive (also in `EXPO_PUBLIC_SUPABASE_URL`). Tracking it (D-03) is the documented decision. No additional mitigation needed. |
| T-02-04 | I | `app/package.json` gen:types script with literal project-ref | accept | project-ref is non-sensitive (D-03). Hard-coding it removes the PowerShell env-var-interpolation footgun (Open Question #4). |
| T-02-05 | E | `npx supabase login` writes a personal access token to `~/.supabase/access-token` | transfer | Token lifecycle is owned by Supabase CLI / OS user account; not Phase 2's surface. Compromise mitigation = `supabase logout` + revoke from Supabase Dashboard. Out of scope for Phase 2 ASVS L1. |
